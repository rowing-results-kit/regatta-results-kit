# SPEC: ハブ機能 v1 — 複数大会を1か所で管理するハブ

作成: 2026-06-12 Fable 5（PM）。Gate1 確定前（PM 裁定待ち）。
「設計選択肢」セクションの A/B/C は PM が裁定してから実装着手すること。

---

## 1. 背景と優先順位

### 現状の課題

現在の `hub/` には `association.json`（3フィールド: id / name / year / dates / venue / status / url）と
`hub/index.html`（大会一覧 HTML）が存在するが、以下が未実装:

- 大会の追加・ステータス変更をブラウザだけで行う手段がない（非エンジニアには `association.json` 直接編集が困難）
- ハブリポジトリをどこに置くか（独立型 vs 同居型）が未定義
- 管理者ポータルとのハブ管理 UI 連携がない
- 大会追加ワークフロー（GHA dispatch フォーム）が存在しない

### ゴール（確定）

> **非エンジニアの協会担当者が、ブラウザ操作だけで大会の追加・ステータス変更・公開できる。**
> 配布者（龍偉）は個別サポートなしで複数大会を同一ハブ下に管理できる。

### 優先順位（変更禁止）

1. **ブラウザ完結** — ターミナル・Python・git コマンド操作を協会側フローから全廃する（オンボーディング v2 の設計思想を踏襲）
2. **追加ステップ最小化** — 既存 7 ステップジャーニーへの挿入を最小限にする
3. **3アカウント超過禁止** — GitHub + Google + Cloudflare の 3 つのみ（オンボーディング v2 と同じ制約）

---

## 2. ジャーニー（既存 7 ステップへの挿入）

### 現在の協会担当者ジャーニー（オンボーディング v2 確定済み）

```
Step 1: テンプレートリポジトリのコピー（GitHub）
Step 2: Cloudflare Pages 接続
Step 3: 大会初期構築（GitHub Actions Run workflow）
Step 4: GitHub PAT 発行
Step 5: GAS コピー作成
Step 6: GAS スクリプトプロパティ設定（PAT + Drive フォルダ）
Step 7: 初期セットアップ実行（管理者ポータル「初期セットアップ実行」ボタン）
```

### ハブ機能の挿入位置

ハブは「最初の大会リポジトリ + ハブ同居型」を採用した場合（選択肢 A-2）、
Step 3 完了後かつ Step 4 着手前に以下を追加する（追加ステップは 1 つに抑える）:

```
Step 3.5（ハブ有効化）: ハブ公開 workflow 実行 → hub/index.html が最初の大会を反映した状態でデプロイ
```

独立型（A-1）の場合は、協会がハブリポジトリを別途作成するため Steps 1〜3 が 2 回発生する。
**追加ステップ最小化の原則から A-2 推奨**（詳細は選択肢 A を参照）。

## 3. 設計選択肢（PM 裁定用）

### 選択肢 A: ハブの置き場所

| | A-1: 独立型 | A-2: 同居型（推奨） |
|---|---|---|
| 説明 | ハブ専用リポジトリを1つ作り、各大会は独立リポジトリ | 最初の大会リポジトリにハブを同居（hub/ ディレクトリが既存） |
| 協会の操作数 | リポジトリ 2 つ以上を管理。Cloudflare Pages も 2 つ以上 | リポジトリ 1 つで完結。Cloudflare Pages も 1 つ |
| 大会追加時 | 新リポジトリ作成 → ハブ JSON に追記 → デプロイ | ハブ JSON に追記 → デプロイのみ（リポジトリ追加不要） |
| メンテ難易度 | リポジトリ数が増えるほど複雑化 | 1 リポジトリで完結。シンプル |
| 推奨 | ✗ 非推奨（非エンジニアには複雑すぎる） | ✅ 推奨 |

**推奨: A-2（同居型）**
根拠: 既存の `hub/` ディレクトリ・`association.json`・`hub/index.html` が同居型前提で作られている。
追加ステップ最小化の原則にも合致。

---

### 選択肢 B: 大会一覧の更新方法

| | B-1: GHA dispatch フォーム（推奨） | B-2: association.json 手編集 |
|---|---|---|
| 操作 | GitHub Actions → Run workflow のフォームで入力 | GitHub 上で JSON を直接編集 |
| 前提スキル | GitHub アカウントのみ（フォームを埋めるだけ） | JSON 記法の理解が必要 |
| エラー耐性 | バリデーションをワークフロー側で実施可能 | 手打ちミス（カンマ・クォート漏れ）でサイト崩壊リスク |
| 既存設計との整合 | setup-tournament.yml と同じ UX（Run workflow） | 既存 7 ステップの設計思想に反する |
| 推奨 | ✅ 推奨 | ✗ 非推奨（非エンジニア前提には不向き） |

**推奨: B-1（dispatch フォーム）**
根拠: 非エンジニア前提かつ既存の `setup-tournament.yml` と同じ操作パターン。
JSON 手編集は1文字ミスでサイトが壊れる。

---

### 選択肢 C: 大会ステータス表現

| | C-1: 3状態（推奨） | C-2: 4状態 |
|---|---|---|
| 状態 | 開催予定 / 速報中 / 結果確定 | 未公開 / 開催予定 / 速報中 / 結果確定 |
| 現状実装 | hub/index.html・association.json に 3 状態（upcoming / live / final）が実装済み | 未実装 |
| 担当者の判断コスト | 3択。明確 | 「未公開」と「開催予定」の区別が曖昧になりやすい |
| ユースケース | 非エンジニアが速報サイトを運用する範囲で十分 | 下書き管理など CMS 的ユースケースが必要な場合に有効 |
| 推奨 | ✅ 推奨（現状実装を流用できる） | ✗ 原則不要。将来要件が出た時点で検討 |

**推奨: C-1（3状態）**
根拠: 既存 `hub/index.html`・`association.json` がすでに 3 状態で実装済み。
追加スキーマ変更ゼロで着手できる。「未公開」状態は担当者の混乱を招くため現時点では不要。

## 5. 受入条件

以下をすべて満たした時点で W5 完了とみなす。

- [ ] 非エンジニアが GitHub Actions の Run workflow フォームだけで大会を追加できる
- [ ] 追加した大会が `hub/index.html` に反映される（Cloudflare Pages 自動デプロイ後）
- [ ] ステータス（upcoming / live / final）を Run workflow フォームで変更できる
- [ ] `association.json` を手編集しなくても運用できる（手編集なしで全操作が完結）
- [ ] 管理者ポータルからハブの大会一覧・ステータスを確認できる（UI）
- [ ] 入力バリデーション: 日付形式・必須フィールド空欄でジョブが意味のあるエラーを返す
- [ ] E2E テスト: 大会追加 → ステータス変更 → hub/index.html での表示確認が手動で一巡できる

## 6. テスト戦略

### 手動テスト（非エンジニアが実施可能なもの）

| # | テスト内容 | 確認方法 |
|---|---|---|
| HT-1 | 大会追加フォームに有効な値を入力して Run workflow → Actions 緑チェック | GitHub Actions タブで確認 |
| HT-2 | HT-1 後に `hub/association.json` に大会が追記されているか | GitHub リポジトリで JSON を確認 |

| HT-4 | ステータス変更フォームで `live` に変更 → バッジが「速報中（赤）」になるか | ブラウザで確認 |

## 7. スコープ外（明示必須）

以下は本 SPEC のスコープ外。将来要件として別 SPEC で扱う:

- **横断検索**: 複数大会をまたいだ選手・タイム検索
- **選手 DB**: 選手情報の一元管理・プロファイルページ
- **過去大会アーカイブの自動移行**: 既存の masters-regatta-2026 等を自動でハブに登録する
- **未公開（draft）状態**: C-2 相当。現時点では不要と判断
- **ハブの自動ステータス更新**: 大会日程から状態を自動判定する仕組み（GAS heartbeat 相当）
- **複数ハブ（協会をまたぐ管理）**: 1 協会 = 1 ハブ前提
